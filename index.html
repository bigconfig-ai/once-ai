once-ai

A highly specific deployment shim for bigconfig.ai. It pins the upstream io.github.bigconfig-ai/once library and feeds it one active ai profile that provisions Oracle Cloud infrastructure, Cloudflare DNS, Resend email, S3-backed OpenTofu state, and the ONCE applications for the bigconfig.ai domain.

Babashka Clojure deps.edn OpenTofu Ansible Oracle Cloud Cloudflare Resend GitHub CLI plugin

Introduction

once-ai is not an application repository. It contains the configuration needed to run an upstream once deployment for bigconfig.ai. The executable logic lives in the once dependency; this repo supplies the active params map, local task names, a small GitHub Actions secret plugin, and tests for that plugin.

The important distinction is:

  • Dependency coordinate: io.github.bigconfig-ai/once, pinned in deps.edn.
  • Upstream namespaces: still io.github.amiorin.once.*, required by bb.edn.
  • Local namespace: io.github.bigconfig-ai.once-ai.plugin, used to add one side effect to the upstream workflow.
Secret policyThis manual intentionally documents variable names and data flow, not live secret values. Do not paste .envrc, API tokens, OAuth credentials, superuser passwords, or cloud access keys into docs, logs, issues, or commits.

Quick Start

1. Install local tooling

Install the base tools in Prerequisites. At minimum you need Babashka, Clojure CLI dependency resolution, OpenTofu, Ansible, SSH, curl, skopeo, OCI CLI, AWS CLI, and GitHub CLI.

bb tasks
bb help

The first invocation may resolve the pinned upstream once git dependency into ~/.gitlibs.

2. Provide secrets and operator credentials

Create a private .envrc from .envrc.example, fill values locally, then allow direnv if you use it.

cp .envrc.example .envrc
# edit .envrc without committing it
direnv allow

Also ensure standard provider credentials are available outside BC_PAR_* where the provider CLIs expect them: OCI CLI config for profile DEFAULT, AWS credentials capable of accessing the OpenTofu state bucket, and an authenticated gh session with permission to set organization Actions secrets.

3. Load the compute private key

The public key in :compute-pubkey must match a private key loaded in ssh-agent. Upstream validation checks this for cloud compute profiles because Ansible needs the agent to reach the new VM.

ssh-add -L
ssh-add ~/.ssh/id_ed25519

4. Validate

bb validate

This runs upstream pre-flight checks for schema, tools, credentials, image reachability, Cloudflare zone presence, and ssh-agent state. It does not fully validate the local GitHub secret plugin; check gh auth status separately.

5. Create / reconcile

bb ai create

The upstream workflow renders templates into .dist/, provisions or reconciles infrastructure with OpenTofu, configures the host with Ansible, and reconciles the configured ONCE applications.

6. Describe

bb describe

Use the upstream describe command to inspect providers, compute reachability, and deployed ONCE applications.

7. Delete

export BC_PAR_COMPUTE_PREVENT_DESTROY=false
bb ai delete
WarnThe upstream compute templates use prevent_destroy by default. Deletion is intentionally blocked until you explicitly override the safety flag.

Safety model

  • Secrets are external. Runtime credentials are read from environment variables and provider CLIs, not from tracked source.
  • Generated output is disposable. .dist/ is rendered by upstream once. Do not hand-edit it; change bb.edn or upstream templates and render again.
  • Compute deletion is gated. Upstream templates default to prevent_destroy = true. Set BC_PAR_COMPUTE_PREVENT_DESTROY=false only when intentionally destroying compute.
  • The local plugin is fail-fast. If GitHub secret inspection fails for a non-404 reason, the plugin throws instead of silently widening secret visibility.
  • Dotfile gitignore. This repository's .gitignore ignores dotfiles by default (.*) and explicitly unignores .gitignore and .envrc.example. Non-dot files such as index.html are not ignored by that rule; new private dotfiles require an intentional unignore strategy.

Dependency model

bb.edn declares one local dependency root:

{:deps {local/deps {:local/root "."}}}

The actual upstream library is pinned in deps.edn:

{:paths ["src"]
 :deps {io.github.bigconfig-ai/once
        {:git/sha "901e266c3d0c9d78408d2ea902ac8bb10f06b3f1"
         #_#_:local/root "../once/main"}}}

This has three consequences:

  1. bb tasks, bb validate, or any other task may fetch the pinned upstream SHA into ~/.gitlibs.
  2. To test against a local upstream checkout, edit deps.edn and switch from :git/sha to :local/root "../once/main".
  3. The upstream task namespaces required by bb.edn use io.github.amiorin.once.*, even though the dependency coordinate is io.github.bigconfig-ai/once.

Delete pipeline

bb ai delete reverses only the OpenTofu-managed resources:

tofu-smtp-post → tofu-dns → tofu-smtp → tofu

Ansible stages are configuration stages, not destroyable resources. Remote host state is effectively removed only when the compute instance is destroyed.

Parameter flow

The active map in bb.edn is passed into upstream workflow code and then repeatedly enriched by environment variables and Tofu outputs.

bb.edn opts
  │
  ├─ :big-config.render/profile = "ai"
  └─ :big-config.workflow/params = static provider + app params
      │
      ▼
package/once* parses CLI args and selects :create / :delete
      │
      ▼
params/once-opts and params/opts-fn
      │
      ├─ workflow/read-bc-pars merges BC_PAR_* environment values
      ├─ tofu-smtp-params reads Resend/SMTP Tofu outputs
      └─ tofu-params reads compute Tofu outputs, including :ip
      │
      ▼
rendered templates + Ansible inventory + local plugin

Environment variables win over static params. A key like :litestream-access-key-id maps to BC_PAR_LITESTREAM_ACCESS_KEY_ID: uppercase the key name and replace hyphens with underscores.

Template placeholders

The upstream renderer uses custom delimiters to avoid HCL conflicts. Application env values in this repo use the filter form:

"LITESTREAM_BUCKET=<{ litestream-bucket }>"
"SUPERUSER_PASSWORD=<{ superuser-password }>"
"GOOGLE_CLIENT_SECRET=<{ google-client-secret }>"

Those placeholders are resolved from ::workflow/params after BC_PAR_* values are merged.

Local plugin: GitHub SERVER_IP secret

The only local source namespace is io.github.bigconfig-ai.once-ai.plugin. It registers a big-config.pluggable/handle-step method for the upstream step keyword :io.github.amiorin.once.tools/ansible-local.

Registration

bb.edn requires the plugin namespace before tasks run:

:requires ([io.github.amiorin.once.describe :as describe]
           [io.github.amiorin.once.package :as package]
           [io.github.amiorin.once.params :as params]
           [io.github.amiorin.once.tools :as tools]
           [io.github.amiorin.once.validation :as validation]
           [io.github.bigconfig-ai.once-ai.plugin])

Execution logic

(defmethod handle-step :io.github.amiorin.once.tools/ansible-local
  [f _step _step-fns opts]
  (let [opts' (f opts)]
    (if (zero? (::bc/exit opts'))
      (update-github-secret opts')
      opts')))

The override first runs the upstream ansible-local stage. If that stage succeeds, it reads :ip from ::workflow/params and updates the org-level GitHub Actions secret.

GitHub API behavior

  1. Read existing secret metadata with gh api /orgs/bigconfig-ai/actions/secrets/SERVER_IP.
  2. If the secret does not exist and GitHub returns 404, default visibility to all.
  3. If the existing secret has visibility = selected, query gh api --paginate /orgs/bigconfig-ai/actions/secrets/SERVER_IP/repositories.
  4. Run gh secret set SERVER_IP --org bigconfig-ai --body <ip> --visibility <visibility>.
  5. For selected visibility, pass --repos with the current selected repository names so the update preserves repository access.
Operator checkRun gh auth status before bb ai create. A missing or under-scoped GitHub login can fail the workflow after infrastructure has already been applied.

Not a delete hook

The plugin updates SERVER_IP during create/reconcile. It does not clear or rotate the secret during bb ai delete.

Active profile

The active profile is an EDN map bound to opts in bb.edn. Upstream tasks receive it as the second argument:

ai {:doc "bb ai create | bb ai delete"
    :task (package/once* *command-line-args* opts)}

The profile contains two top-level entries:

KeyValueMeaning
:big-config.render/profile"ai"Render/profile name used by upstream output paths and template selection.
:big-config.workflow/paramsNested params mapProvider selection, provider config, domain, package, SSH keys, and ONCE applications.

Task inventory

bb tasks exposes the public task set:

help     the workflow help
ai       bb ai create | bb ai delete
validate validate active profile, tools, credentials, and docker images
describe describe active profile providers, compute, and deployed applications

There are also internal/scoped tasks whose names begin with -; they are useful for tests, formatting, and individual workflow stages.

bb ai

The primary entry point delegates directly to upstream package/once* with the active opts map.

bb ai create           # render and apply the full create pipeline
bb ai delete           # destroy OpenTofu-managed resources in reverse order
bb ai delete create    # clean-slate redeploy pattern

Create effects

  • Renders upstream templates under .dist/.
  • Initializes and applies OpenTofu stages.
  • Reads Tofu outputs back into workflow params.
  • Updates local SSH config via ansible-local.
  • Runs the local GitHub secret plugin after successful ansible-local.
  • Configures the remote host and reconciles ONCE applications via ansible.

Delete effects

  • Destroys tofu-smtp-post, tofu-dns, tofu-smtp, and tofu in that order.
  • Does not run Ansible destroy hooks.
  • Does not clear GitHub SERVER_IP.

bb validate

Runs upstream validation for the active params. It is the standard pre-flight before any create or destructive operation.

  • Schema: provider-specific params exist and have the expected shapes; app hosts are valid and within bigconfig.ai.
  • Tools: required CLIs are present on PATH.
  • Credentials: provider tokens and local CLI credentials work against live APIs where upstream supports checks.
  • SSH agent: cloud compute profile requires the configured compute public key to be loaded in ssh-agent.
  • Images: every :image resolves via skopeo inspect.
bb validate
Scopebb validate is upstream validation. The local GitHub secret plugin has unit tests, but validate does not guarantee that gh secret set will succeed at deployment time.

bb describe

Delegates to upstream io.github.amiorin.once.describe/describe*. It renders/merges params and reports provider configuration, compute reachability, and deployed application state.

bb describe

Use this after bb ai create or after changing image tags to confirm the running state. Upstream describe can compare running image digests to live registry digests when the remote host is reachable and registry inspection succeeds.

Deploy runbook

  1. Sync source: update this checkout and confirm deps.edn pins the intended upstream once SHA.
  2. Load environment: ensure .envrc exports all variables from .envrc.example; do not print values.
  3. Check cloud CLIs: oci, aws, and gh should already be authenticated for the target accounts/org.
  4. Check SSH agent: ssh-add -L contains the private half for :compute-pubkey.
  5. Pre-flight: run bb validate and fix every issue.
  6. Apply: run bb ai create.
  7. Inspect: run bb describe.
  8. Verify externally: open www.bigconfig.ai, bigconfig.ai, forms.bigconfig.ai, and marketplace.bigconfig.ai.
  9. Check GitHub secret: verify that org Actions secret SERVER_IP was updated for the intended visibility/repositories.

Teardown runbook

  1. Confirm the teardown is intentional and that backups/application data are handled separately.
  2. Ensure all provider credentials are still valid: AWS state access, OCI, Cloudflare, Resend.
  3. Export the destroy override only for the delete command:
BC_PAR_COMPUTE_PREVENT_DESTROY=false bb ai delete
  1. Review OpenTofu output for all four destroy stages.
  2. Manually decide whether to rotate or remove GitHub SERVER_IP; the workflow does not remove it.
DangerDeleting compute removes the host running ONCE applications. Marketplace persistence depends on the configured application-level backup/replication story, not on this repo.

Update runbook

Update an application image

  1. Edit the relevant :image in bb.edn. Prefer immutable tags or digests when release reproducibility matters.
  2. Run bb validate so skopeo verifies the image exists.
  3. Run bb ai create to reconcile the ONCE app definition.
  4. Run bb describe to compare the running digest with the registry digest.

Update upstream once

  1. Change the :git/sha in deps.edn.
  2. Run bb tasks to force dependency resolution.
  3. Review upstream release/source changes, especially workflow step names and handle-step signature compatibility.
  4. Run bb -test for local plugin tests.
  5. Run bb validate.

Change provider-level infrastructure

Changing OCI subnet, compartment, shape, boot volume, or domain/DNS values may cause Tofu to replace resources. Always inspect plans through individual stage tasks before applying high-impact changes.

Troubleshooting

bb tasks cannot resolve dependencies

  • Check network access to GitHub.
  • Confirm deps.edn contains a valid :git/sha.
  • Remove or refresh stale entries in ~/.gitlibs only if dependency resolution is clearly corrupted.

bb validate fails on tools

  • Install the missing command reported by validation.
  • Ensure it is available in the same shell where direnv/environment variables are loaded.

OCI validation or apply fails

  • Confirm ~/.oci/config has profile DEFAULT.
  • Confirm the configured subnet, compartment, and availability domain still exist and are accessible.
  • Oracle free-tier/ARM capacity can be region and AD dependent; shape creation may fail despite valid config.

S3 backend fails

  • Do not confuse the OpenTofu state bucket with Litestream application storage.
  • Run an AWS identity/bucket check with the same environment used for bb ai create.
  • Confirm the bucket region is eu-west-1 and credentials permit state read/write/lock behavior required by OpenTofu.

Cloudflare or Resend fails

  • Confirm BC_PAR_CLOUDFLARE_API_TOKEN has access to the bigconfig.ai zone.
  • Confirm BC_PAR_RESEND_API_KEY is valid for the Resend account.
  • DNS propagation can delay Resend verification after the DNS stage.

Ansible cannot connect

  • Run ssh-add -L and verify the configured compute public key is loaded.
  • Check the local SSH config generated by ansible-local.
  • Confirm the VM has a public IP from the Tofu output and security rules allow SSH.

Marketplace app starts without expected config

  • Check every BC_PAR_LITESTREAM_*, BC_PAR_SUPERUSER_*, and BC_PAR_GOOGLE_* variable is exported.
  • Inspect rendered Ansible/ONCE output in .dist/ without committing secrets.

GitHub secret update fails

  • Run gh auth status.
  • Confirm the authenticated GitHub identity can manage organization Actions secrets for bigconfig-ai.
  • If the existing secret uses selected repositories, confirm the identity can read that selected repository list.
  • Rerun the failed workflow once auth is fixed; upstream stages are designed to be idempotent.

bb ai delete refuses to destroy compute

This is expected. Set BC_PAR_COMPUTE_PREVENT_DESTROY=false only for the delete operation.
